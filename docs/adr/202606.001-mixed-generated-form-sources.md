# ADR 202606.001: Mixed generated and handwritten form sources

Status: Accepted

Date: 2026-06-19

## Context

Generated `@SwingForm` support currently targets non-generic records. This is
the right compact path for immutable edit data, generated field keys,
descriptors, form models, view contracts, behavior plans, and bindings.

Practical Swing forms are not always pure generated records. Some forms need
handwritten methods, derived values, helper behavior, or domain-specific
presentation logic that should live with the form source. Applications should
not have to abandon generated metadata and typed companions merely because one
part of a form is handwritten.

This decision depends on the generated-companion philosophy used across
Corusco: generated keys, descriptors, models, and bindings should remain the
normal integration path, not handwritten string ids or runtime annotation
scanning.

## Decision

Extend `@SwingForm` to support abstract classes in addition to records.
Record-based forms remain the compact immutable data-carrier path. Abstract
class forms support the AutoValue-style case where a developer wants generated
metadata and model wiring but also needs handwritten domain or presentation
logic.

A valid abstract-class form source:

- is an abstract class annotated with `@SwingForm`;
- declares annotated abstract accessor methods as generated fields;
- may declare concrete handwritten methods;
- does not rely on runtime annotation scanning;
- produces generated companions using the same typed key and descriptor
  philosophy as record forms.

For the first implementation, the abstract-class source contract is deliberately
strict:

- the source must be non-generic, matching the current record restriction;
- generated fields are declared directly on the annotated class as abstract,
  no-argument methods;
- field names are the method names themselves, for example `name()` becomes the
  `name` field;
- JavaBean-style normalization, inherited annotated methods, overloaded
  accessors, and generated implementations of unrelated abstract methods are
  out of scope;
- any unannotated abstract method on the annotated class is a processor error;
- field order follows source declaration order as reported by the compiler model.

Field-kind, validation, and help annotations that currently target record
components must also support abstract accessor methods. In practice, the
relevant source-retained annotations should target both `RECORD_COMPONENT` and
`METHOD`, and processor diagnostics should continue to point at the annotated
field source.

Example:

```java
@SwingForm(id = "customer")
public abstract class CustomerEdit {
    @TextField
    @Required
    public abstract String name();

    @ComboBox
    public abstract CustomerType type();

    public boolean corporateCustomer() {
        return type() == CustomerType.CORPORATE;
    }
}
```

The processor should generate a concrete immutable result type for the abstract
source, for example `GeneratedCustomerEdit`. The generated result type should
be final, extend the abstract source, implement all generated field accessors,
and provide deterministic value semantics (`equals`, `hashCode`, and
`toString`) comparable to records.

The generated form model should commit to the abstract source type, not to the
generated implementation type:

```java
public final class CustomerEditFormModel
        extends AbstractFormModel<CustomerEdit> {
    @Override
    protected CustomerEdit createResult() {
        return new GeneratedCustomerEdit(name.value(), type.value().value());
    }
}
```

Generated field keys and descriptors should likewise use the abstract source
type as owner type. The generated implementation class is part of the generated
API and must have a stable name, but ordinary application code should be able
to depend on the source abstraction and generated companions rather than on the
implementation class.

The exact generated implementation naming convention should be decided before
implementation and then treated as generated API. Generated names should avoid
colliding with the existing companions (`CustomerEditFields`,
`CustomerEditFormModel`, `CustomerEditView`, and similar).

Internally, the processor should stop treating record components as the only
form-field source. It should normalize records and abstract accessor methods
into a source-independent form specification before writing generated sources.

Concrete methods in the abstract source are result-level behavior. They operate
on committed result values returned by the generated accessors. They are not
live form-model extension hooks and should not be described as observing
intermediate edit state.

## Consequences

Generated form usage becomes practical for larger screens where some pieces
must stay handwritten.

The annotation processor keeps Corusco's generated-companion philosophy while
becoming less tied to Java records.

Abstract-class form generation introduces generated type naming and inheritance
decisions that become public API once released.

Processor complexity will increase. A normalized form-source model should be
introduced before adding many new annotations or source-type variants.

Field annotations become usable on both record components and abstract methods.
That expands the annotation API surface and requires documentation updates in
the annotations package, generated-form package docs, and processor diagnostics.

The first abstract-class implementation intentionally rejects some Java shapes
that could be supported later, such as inherited fields or JavaBean-style
accessors. This keeps generated ids, descriptor order, and owner typing
predictable while the source model is generalized.

## Out of Scope

This decision does not introduce live form-model customization hooks. If a
screen needs behavior that observes intermediate edit state, that should be
handled by generated companions, bindings, or a later form-model extension
decision rather than by concrete methods on the committed result source.

This decision also does not introduce annotation-driven parent/sub-form
composition. Multi-form dialog sessions are covered separately by ADR
202606.003.

## Implementation Notes

1. Introduce an internal normalized form-source model while preserving current
   record output.
2. Expand relevant form, validation, and help annotation targets from record
   components to record components plus methods.
3. Add validation for abstract-class source forms, including non-generic source
   type, abstract no-argument accessor shape, duplicate field names/ids,
   unsupported field types, unannotated abstract methods, and generated-name
   collisions.
4. Generate concrete immutable result types for abstract-class forms.
5. Generate fields, descriptors, models, view contracts, behavior plans, and
   bindings from the normalized source model.
6. Add generated-source tests covering annotation targets, diagnostics, result
   typing, generated implementation value semantics, and one small example using
   handwritten result-level methods.
