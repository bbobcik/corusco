# ADR 202606.005: Fixed-schema time-series and tidy data

Status: Accepted

Date: 2026-06-27

## Context

ADR 202606.004 added technology-neutral enterprise data support for windowed
table results, row identity, key-based selection, request lifecycle, and edit
staging. That decision deliberately deferred time-series and tidy-data APIs
because they have additional invariants around time axes, measured values,
missing data, quality flags, aggregation, and dense storage.

Corusco already has generated table metadata through `@CoruscoTable` and
`@Column`. That path is useful when a record is naturally a Swing table row:
generated code creates typed table keys, column descriptors, resource keys,
row accessors, row updaters, and Swing table bindings. The current table path
is row-oriented and presentation-oriented. It describes how a row is displayed
or edited in a table, not the semantic shape of a fixed analytical data set.

Some application data is better described as a fixed, known schema:

- a time-indexed set of observations with a timestamp, dimensions, measured
  values, units, quality flags, and missing values;
- a tidy data table where every row is one observation and every variable is
  an explicitly typed column;
- a dense sensor, market, operational, telemetry, or forecasting result where
  primitive values should not require per-cell boxing or per-row object
  allocation;
- a data extract that should be rendered in a table, chart, validation view,
  or export pipeline from the same generated schema.

The previous assessment recommended adding time-series and tidy-data
primitives: column schema, time axis, unit metadata, quality and missing-value
semantics, and aggregation/downsampling request models. This ADR narrows that
recommendation to data whose structure is known at compile time. Runtime,
schema-on-read data frames are useful, but they do not match Corusco's current
strength: javac annotation processing, typed generated companions, stable
metadata, and direct Java access without reflection.

The goal is not to turn Corusco into a full analytics engine. The goal is to
make these concepts explicit enough that applications, Swing tables, charts,
exports, validators, and future adapters can share a compact, typed model
instead of passing raw lists, maps, arrays, or backend-specific result objects.

## Intent

Add a fixed-schema data description and generated access layer for
time-series and tidy data.

The model should let application code declare the semantic shape once and then
consume generated descriptors, accessors, frame builders, and optional table
or chart adapters. The generated layer should make hot paths allocation-aware:
primitive columns should be stored as primitive arrays where possible, missing
and quality state should be compact, and row objects should be optional views
rather than the primary storage format for dense data.

The feature should integrate with existing Corusco concepts:

- generated typed keys and descriptors instead of arbitrary string ids;
- `corusco-core` as the Swing-free home of semantic contracts;
- `corusco-annotations` and `corusco-processor` as the compile-time API and
  generation layer;
- optional bridges to `core.table`, `core.data`, and `corusco-swing` instead
  of a new parallel table framework;
- generated-source tests, package docs, examples, and manual text when the
  API becomes public.

## Decision

Introduce a new semantic data-set contract instead of extending
`@CoruscoTable` or `@Column`.

`@CoruscoTable` and `@Column` remain presentation-table metadata. They carry
table ids, resource keys, default widths, visibility, sort/filter capability,
persistence ids, and editability. Time-series and tidy-data metadata has a
different job: it names semantic columns, roles, units, time axes, missingness,
quality semantics, and aggregation capabilities. A data-set descriptor may be
adapted to a table descriptor, but it is not itself a table descriptor.

The source annotation should describe a fixed schema. The exact names can
change during implementation, but the intended shape is:

```java
@CoruscoDataSet(id = "market/quotes")
public record MarketQuote(
        @DataColumn(role = DataColumnRole.SEQUENCE)
        long sequence,

        @TimeAxis(unit = TimeUnit.MILLIS, monotonic = true)
        long timestampMillis,

        @Dimension
        String symbol,

        @Dimension
        String venue,

        @Measure(unit = "USD", missing = MissingPolicy.NAN)
        double bid,

        @Measure(unit = "USD", missing = MissingPolicy.NAN)
        double ask,

        @Measure(unit = "shares")
        long volume,

        @QualityColumn(appliesTo = {"bid", "ask", "volume"})
        int quality
) {
}
```

The generated companions should expose semantic data descriptors rather than
only display columns. A representative generated surface is:

```java
MarketQuoteDataColumns.TIMESTAMP_MILLIS
MarketQuoteDataColumns.BID
MarketQuoteDataDescriptor.DESCRIPTOR
MarketQuoteFrame.builder(expectedSize)
MarketQuoteFrameTableAdapter.descriptor()
```

The descriptor package lives in `corusco-core` under
`cz.auderis.corusco.core.dataset`. The package stays Swing-free and must not
depend on the processor. The small enum vocabulary that is used directly in
annotation attributes, such as `DataColumnRole`, `MissingPolicy`,
`QualityPolicy`, and `AggregationFunction`, lives in `corusco-annotations`.
Core descriptors reuse those types so source annotations and runtime
descriptors cannot drift apart.

## Core Model

The core semantic model should describe stable facts:

- data-set key: stable id and row/source type;
- column key: stable id, owning data-set key, semantic role, Java value type,
  and physical storage type;
- column role: time axis, dimension, measure, quality, sequence, metadata, or
  auxiliary value;
- time axis: column key, temporal representation, epoch unit where relevant,
  monotonic/sorted hints, duplicate timestamp policy, and display-zone
  metadata only where it is stable schema information;
- units: stable unit id or symbol, optional display resource key, and optional
  quantity kind such as price, energy, volume, count, ratio, duration, or
  latency;
- missing-value semantics: null, NaN, sentinel, bitset, always-present, or
  application-defined policy;
- quality semantics: compact flags or typed quality values, declared separately
  from missingness;
- aggregation metadata: allowed aggregation functions for a measure and the
  default quality/missingness behavior during aggregation;
- data-set descriptor: ordered columns, optional time axis, dimensions,
  measures, quality mappings, and declared capabilities.

The model should distinguish missingness from quality. A missing value means
there is no value for that observation. A poor-quality value may still exist
and may be displayed, filtered, aggregated, or exported according to policy.

The first unit model should be deliberately small. A stable unit id and
resource/display hook are enough for Corusco's core. A full dimensional
analysis library, unit conversion engine, or JSR-385 dependency should be
deferred until real adapter requirements justify it.

## Generated Storage

The generated storage layer should be optional but central to the performance
story.

For fixed schemas, generated code can produce a frame-like columnar container:

- primitive arrays for primitive measure, time, sequence, and quality columns;
- object arrays for dimensions or object-valued columns;
- compact bitsets for missingness when the missing policy requires a separate
  presence marker;
- generated append/build operations that validate column lengths and required
  values;
- generated row views only when code needs the source record shape;
- generated direct column accessors for table, chart, export, and validation
  adapters.

The generated frame is not a replacement for every row-list use case. Small
screen tables can still use generated `@CoruscoTable` records and
`ObservableTableModel`. Dense analytical data should not be forced through a
list of row records merely to render or aggregate it.

The first implementation may generate descriptors before generating storage.
That keeps the public semantic model reviewable before committing to a
specific frame API. Once descriptors are stable, generated storage should be
added as a separate slice with its own tests and examples.

## Requests and Aggregation

Core should express aggregation and downsampling requests, not implement a
complete analytics engine in the first slice.

Useful request records include:

- time range, expressed against a declared time axis;
- dimension filters using generated column keys;
- requested measures;
- bucket width or target sample count;
- aggregation function per measure, such as first, last, min, max, sum, count,
  average, weighted average, or application-defined function id;
- missing-value policy for each measure or request;
- quality policy, such as require-good, include-with-flag, worst-quality,
  best-quality, count-by-quality, or application-defined policy id;
- alignment policy for buckets, such as epoch-aligned, range-start-aligned, or
  calendar-aligned where a time zone is explicit;
- sort/order expectations.

Adapters and applications can execute those requests against SQL, time-series
databases, in-memory frames, files, or services. Core should validate the
request shape where it can, for example by rejecting average aggregation over
a dimension column or a time bucket request without a time axis. It should not
pretend that every backend supports every aggregation function.

## Bridges

The data-set layer should bridge to existing Corusco table concepts without
making table rendering the primary model.

Potential bridges:

- generate a read-only `TableDescriptor` view over selected data-set columns;
- generate a Swing table model over a generated frame without materializing
  row records;
- translate data-set column metadata to table headers, tooltips, and resource
  keys;
- expose `CoruscoDataQuery` terms for dimensions and time ranges where that
  mapping is meaningful;
- provide chart-friendly accessors in a later adapter without adding chart
  APIs to core.

The bridge must preserve semantic metadata. A table column created from a
measure should still know its unit, missingness policy, and quality mapping
through the data-set descriptor. Table state persistence remains a table
concern and should use table persistence ids rather than changing data-set
column identity.

## Annotation Processing

The processor should follow the existing scanner/spec/writer pattern used by
forms, actions, and tables.

The first processor stage should:

- validate that the data-set source is a non-generic record;
- validate stable data-set and column ids;
- classify annotated components into roles;
- require exactly one time axis for time-series descriptors and no more than
  one primary time axis for tidy descriptors unless a later ADR adds multi-axis
  support;
- reject unsupported component types early;
- validate missing-value policies against component types;
- validate quality-column references;
- generate descriptor companions and resource keys;
- avoid Swing-specific output unless an explicit adapter package requests it.

The generated code should stay readable. A developer should be able to open a
generated `MarketQuoteDataDescriptor` or `MarketQuoteFrame` and understand the
schema, accessors, and storage strategy without runtime annotation scanning.

Adapter-package generation should remain possible. Like form and table
metadata, data-set annotations should use class-file retention if future
adapter packages need to generate Swing, chart, or export companions from
already compiled model classes.

## Consequences

Corusco gains a typed way to describe analytical data without making
applications choose between raw arrays and presentation-table row records.

The feature reinforces Corusco's compile-time generation strategy. Fixed
schemas can produce direct accessors, primitive storage, descriptor constants,
and compile-time diagnostics instead of reflection, maps, or JavaBeans
property paths.

The public API surface grows beyond Swing presentation metadata. This is
acceptable only if the layer stays small, semantic, and adapter-friendly. It
should not absorb database connectors, chart libraries, full unit conversion,
or a general data-frame expression language.

Existing table APIs remain valuable. They are the bridge for grid display,
state persistence, selection, validation decoration, and Swing rendering.
They should not become the schema authority for units, quality, missingness,
or time-axis semantics.

Generated storage introduces compatibility obligations. Frame method names,
column key names, descriptor shapes, and missing/quality policies become part
of the generated-source contract once released. Tests must cover both runtime
behavior and generated source shape.

## Alternatives Considered

Extending `@Column` with time-series attributes was rejected. The annotation
already represents table presentation and editing metadata. Adding unit,
quality, aggregation, and missing-value semantics would mix two different
contracts and make ordinary UI tables carry analytical concepts they do not
need.

Using only `core.data` requests was rejected. ADR 202606.004 describes
windowed enterprise result sets and backend query lifecycle. It does not know
column roles, units, time axes, quality mappings, or columnar storage.

Using a generic runtime data-frame API was deferred. Runtime schemas are useful
for imported or ad-hoc data, but they would not exploit Corusco's existing
annotation-processing strengths and would make compile-time diagnostics much
weaker.

Adding a full analytics engine was rejected. Aggregation and downsampling
request models are enough for Corusco core. Execution belongs in adapters,
applications, or later focused modules.

Depending on a full units library in the first slice was rejected. Unit
identity and display metadata are necessary; conversion and dimensional
analysis can be added later if real applications need them.

## Out of Scope

This decision does not define a general-purpose runtime data frame for unknown
schemas.

This decision does not add JDBC, REST, Kafka, file-format, time-series database,
or chart-library adapters.

This decision does not implement live streaming, backpressure, retention
windows, or subscription protocols. Those need their own decision because they
change lifecycle and threading expectations.

This decision does not add a complete analytics expression language or a
general query optimizer.

This decision does not require existing `@CoruscoTable` users to migrate.
Ordinary generated table records remain the recommended path for row-oriented
UI tables.

This decision did not require generated frame storage in the first commit.
The accepted implementation includes both descriptor generation and an initial
generated columnar frame because the descriptor API stabilized enough during
the implementation slice.

## Implementation Plan

1. Add package-level design docs for the chosen core package name and define
   the descriptor vocabulary: data-set key, data column key, column role, time
   axis, unit metadata, missing policy, quality policy, aggregation function,
   aggregation capability, and data-set descriptor.
2. Add focused core tests for descriptor construction, type validation,
   missing/quality metadata, time-axis rules, equality, and stable ids.
3. Add annotations for the first fixed-schema source shape, with class-file
   retention and package docs that explain the separation from `@CoruscoTable`.
4. Add processor scanner/spec support and generated descriptor companions for
   non-generic records.
5. Add generated-source tests for descriptor names, column roles, time-axis
   metadata, unit metadata, missing policies, quality-column references, and
   diagnostics.
6. Add request model records for time-range, aggregation, and downsampling
   requests once descriptors can validate request shape.
7. Add generated columnar frame storage after descriptor APIs stabilize.
8. Add a table bridge that can display a generated frame without forcing row
   materialization.
9. Convert or add a time-series showcase example that demonstrates generated
   descriptors, generated storage, missing/quality metadata, and a table view.
10. Update public API review metadata and generated API Javadocs together with
    the public API. Broader manual chapters and guide-style narrative can
    follow once the feature graduates beyond the initial accepted slice.

## Acceptance Criteria

The first accepted implementation should prove that a fixed-schema
time-series/tidy source can:

- compile through the annotation processor without reflection;
- expose generated typed column keys and a data-set descriptor;
- declare one time axis and validate its type and id;
- distinguish dimensions, measures, sequence columns, and quality columns;
- represent units with stable metadata;
- represent missingness separately from quality;
- reject contradictory or unsupported annotation combinations at compile time;
- define aggregation/downsampling request models without executing them in
  core;
- optionally display through a table bridge while preserving semantic
  descriptor metadata;
- keep Swing-specific generation optional;
- include tests, examples, and docs that make the intended usage clear.
