# ADR 202606.004: Enterprise data support in core

Status: Accepted

Date: 2026-06-27

## Context

Corusco already has table descriptors, observable values, validation models,
and generated form support, but it did not have a technology-neutral model for
large enterprise result sets. Applications that page remote data, sort and
filter through backends, stage edits, or keep selection across changing table
windows would otherwise need to invent those contracts in each adapter or
application.

The existing table APIs are intentionally view-oriented. They describe columns,
table state, and observable row collections, but they do not define stable row
identity, large-result selection, request lifecycle, total-count semantics,
server-side query descriptors, or CRUD staging. Practical enterprise
applications need those concepts before they can bind tables to SQL, REST,
message-backed stores, in-memory services, or other data providers without
coupling core to one transport.

Index-based selection is not sufficient for this use case. Windowed tables may
replace the visible page when a user scrolls, refreshes, sorts, or filters.
Selections must therefore be keyed by stable row identity rather than by current
view position. The model also needs scalable "select all matching the current
query" behavior, because selecting a large backend result set must not require
materializing every row key in memory.

CRUD support has the same boundary problem. Core should expose a way to stage
creates, updates, deletes, conflicts, version tokens, validation problems, and
save results, but it should not own JDBC, REST, Kafka, repository interfaces, or
application-specific persistence workflows. Applications and future adapter
modules can translate these neutral edit records to their own storage systems.

The first slice should stay deliberately narrower than a complete data platform.
Time-series and tidy-data APIs, live feed contracts, and backpressure semantics
are important future work, but they have different invariants from ordinary
enterprise table windows. Adding them now would make the initial core surface
less stable and harder to validate.

Corusco targets Java 25, but this decision must use non-preview facilities only.
The implementation should remain allocation-conscious: immutable snapshots,
primitive counts and generations, expected-size maps and sets where useful,
reset-style page replacement, and explicit nullable state rather than
`Optional` in hot model objects.

## Decision

Add enterprise data support to `corusco-core` as a small set of public,
technology-neutral contracts under `cz.auderis.corusco.core.data` and
`cz.auderis.corusco.core.data.edit`.

The `core.data` package defines immutable read-model primitives:

- `CoruscoRowIdentity<R, K>` extracts stable row keys and records row/key type
  metadata.
- `CoruscoDataRange`, `CoruscoDataCount`, `CoruscoDataQuery`,
  `CoruscoDataSort`, `CoruscoDataFilter`, and
  `CoruscoDataFilterOperator` describe window ranges, count precision, and
  transport-neutral query terms.
- `CoruscoDataRequest` and `CoruscoDataPage<R>` carry load requests and
  immutable page snapshots.
- `CoruscoDataStatus`, `CoruscoDataPhase`, and `CoruscoDataError` describe
  lifecycle state and failures without assuming a backend technology.

`CoruscoDataCount` uses primitive fields to represent unknown, estimated, and
exact counts. Repeated request, page, count, range, and status objects avoid
boxed numeric state in their public representation.

`CoruscoDataLoader<R>` is a synchronous loader invoked with a
`CoruscoDataRequest` and `CancellationToken`. It is intentionally not a
repository abstraction. A loader may delegate to SQL, REST, Kafka-backed
storage, generated test data, or in-memory services, but those choices remain
outside core.

`CoruscoDataSource<R, K>` is the observable presenter-owned read model. It owns
the current immutable window rows, range, query, total count, status, refresh
generation, and lifecycle. `DefaultCoruscoDataSource<R, K>` provides the
standard implementation backed by the existing `TaskService`. It submits loads
as background tasks, keeps the previous page visible while a refresh is
running, increments a generation counter for each request, ignores stale
completions, records failures in status, and propagates `ChangeOrigin`.

Page replacement is modeled as one reset-style collection update. The backing
window list is replaced with an immutable snapshot instead of clearing and
adding individual rows one at a time. This keeps observable change volume
bounded by the number of page replacements, not by the number of rows loaded.

Selection is key-based. `CoruscoRowSelection<K>` and
`CoruscoRowSelectionMode` support:

- `NONE`, for an empty selection;
- explicit included keys, for ordinary row selections;
- all rows matching the current query with explicit excluded keys, for scalable
  select-all behavior.

Selection also carries an explicit lead key. It never interprets the current
view index as identity. Visible-window retention operations work by key and
therefore survive sorting, filtering, and page replacement.

The `core.data.edit` package defines CRUD staging primitives:

- `CoruscoEditOperation`, `CoruscoVersionToken`, and
  `CoruscoEditChange<R, K>` describe staged create, update, delete, and
  validation state.
- `CoruscoEditSet<R, K>` is an immutable stable-order snapshot of staged
  changes.
- `CoruscoEditSession<R, K>` owns mutable staging state, exposes observable
  dirty state and edit-set snapshots, and supports create, update, delete,
  discard, and clear operations.
- `CoruscoSaveResult<R, K>` and `CoruscoConflict<R, K>` model persistence
  outcomes without owning a persistence mechanism.

The edit session stores staged changes in stable-order maps keyed by row
identity. It emits a new immutable snapshot only when state actually changes,
so observers do not need to diff identical edit sets.

The table bridge remains descriptor-based. `CoruscoTableDataDescriptor<R, K>`
binds a `TableDescriptor<R>` to a `CoruscoRowIdentity<R, K>`.
`CoruscoTableDataQueries` maps `TableState` sort metadata to
`CoruscoDataQuery` sort terms through stable column persistence ids. Columns
without persistence ids are not translated into backend query terms.

Core does not introduce Swing `TableModel`, `JTable`, MQTT, Kafka, JDBC,
repository implementations, or backend adapters in this decision. Those belong
in later adapter modules or application code.

## Consequences

Corusco core now has a durable public boundary for enterprise table data that
can be reused by Swing, generated companion code, and future non-Swing adapters.
The boundary describes data requests, visible windows, lifecycle state,
selection, and edit staging without requiring one storage or UI technology.

Large selections are representable without allocating every matching key. The
tradeoff is that consumers must understand the selection mode and apply the
query plus exclusion keys when executing bulk actions.

Window refresh behavior is predictable for presenters. Existing rows remain
visible while a refresh runs, stale task completions are suppressed, and failed
loads update status instead of implicitly clearing the previous page.

Backend adapters are responsible for translating neutral query, filter, sort,
count, edit, conflict, and validation records to their own protocols. Core does
not attempt to validate that every backend can express every filter operator.

The published API surface expands with two exported packages. Package Javadocs,
API review metadata, and manual text must stay synchronized with the new public
contracts.

## Alternatives Considered

Extending only the existing observable table model was rejected. It would make
windowing, backend query metadata, stable identity, and edit staging depend on a
view-facing abstraction that was not designed for remote or large data sets.

Using current row index as selection identity was rejected. Index identity
breaks as soon as a table is sorted, filtered, refreshed, or virtualized.

Materializing all selected keys for select-all behavior was rejected. It is not
viable for large backend result sets and would force user-interface actions to
load data they do not need to display.

Adding JDBC, REST, Kafka, or repository contracts to core was rejected. Those
interfaces would prematurely choose persistence and transport shapes for an API
that should first define presenter-visible behavior.

Adding time-series, tidy-data, live feed, or backpressure contracts in the same
slice was deferred. Those domains have additional ordering, cadence, retention,
and streaming concerns that deserve separate decisions.

## Follow-up Work

Future work can add Swing and generated-companion adapters that consume these
core contracts, including virtualized table models and richer presenter
examples.

Storage-specific adapter modules can translate the neutral data and edit
contracts to SQL, REST, message-backed services, or in-memory implementations.

Filter capability negotiation, richer typed filter values, bulk-action
execution helpers, conflict-resolution user interfaces, and live feed
contracts should be designed as separate decisions once their use cases are
ready.
