# ADR 202606.000: Observable readable collections and sorted sets

Status: Proposed

Date: 2026-06-19

## Context

Corusco currently uses `ObservableList` as the main observable collection
contract. That works well for Swing list, combo-box, and table adapters because
those adapters need indexed reads and list-coordinate change events.

Some presentation views need less. They only need to observe ordered contents,
take snapshots, and refresh when contents change. For these consumers, forcing
every source through `ObservableList` leaks indexed mutation semantics into
types that are not lists.

One important case is an ordered collection of unique elements. Applications
often hold data naturally as a `SortedSet` or as a set with a defined
comparator. They should be able to expose that data to Corusco views without
maintaining a copied list beside the set.

## Decision

Add a common read-only observable collection contract in `corusco-core`:

```java
public interface ObservableReadableCollection<E> {
    int size();
    List<E> snapshot();
    Subscription subscribe(CollectionChangeListener<E> listener);
}
```

The exact listener type is still open, but it should describe observable
content changes without requiring callers to understand list mutation methods.
`ObservableList<E>` should extend this interface.

The name is intentionally conservative but needs careful documentation:
although it says "collection", Corusco read-only views still need a stable
presentation order. `snapshot()` order is meaningful. For an `ObservableList`,
it is list order. For an `ObservableSortedSet`, it is sorted-set iteration
order.

Add a set-first collection type:

```java
public final class ObservableSortedSet<E>
        implements ObservableReadableCollection<E> {
    boolean add(E element);
    boolean remove(E element);
    boolean contains(E element);
    Comparator<? super E> comparator();
    ObservableList<E> asList();
}
```

`ObservableSortedSet` owns sorted unique storage. It should not implement
`ObservableList` directly because indexed insertion, indexed replacement, and
move operations do not have honest set semantics. Instead, `asList()` returns a
stable read-only `ObservableList` view whose indices are derived from the
current sorted order.

This creates two read-only levels:

- `ObservableReadableCollection` for read-only widgets or presenters that only
  need current contents, stable ordered snapshots, and coarse content-change
  notifications;
- `ObservableList` for adapters that require indexed reads and list-coordinate
  changes, such as Swing list, combo-box, and table models.

Corusco should not force all read-only consumers through `ObservableList`
merely because the current adapters are list-based. At the same time, it should
not weaken `ObservableList` by pretending that every observable collection can
support `get(index)`, indexed removal, replacement, or move.

Uniqueness follows the backing sorted-set equivalence rule. When a comparator
is used, two values that compare as equal are the same set element even if
`equals` would distinguish them. This must be documented because it affects
selection models, duplicate suppression, and table-row identity.

The common readable-collection listener may begin with reset-style or
content-changed delivery. The `asList()` view emits changes in list
coordinates. Implementations may begin with reset-style list delivery for
complex reorder operations, matching the existing conservative behavior of
sorted list views, and later optimize insert/remove changes where the index can
be computed precisely.

## Consequences

Read-only views can consume observable lists and sorted sets through a common
type when they only need ordered contents and change notification.

Swing list, combo-box, and table adapters can still consume ordered unique data
as an `ObservableList` view without maintaining a copied list beside a set.

`ObservableReadableCollection` must not become a vague catch-all. Its contract
needs to say whether ordering is meaningful, what event granularity is
guaranteed, and when callers must still use `ObservableList`.

`ObservableSortedSet` must document comparator-based uniqueness clearly or
users will confuse it with `equals`-based set identity.

## Implementation Notes

1. Add `ObservableReadableCollection` and readable-collection change events.
2. Adapt `ObservableList` to extend the readable contract.
3. Add `ObservableSortedSet` with comparator uniqueness tests.
4. Add a read-only `ObservableList` view from `ObservableSortedSet`.
5. Add tests covering readable-collection events and list-coordinate events.
