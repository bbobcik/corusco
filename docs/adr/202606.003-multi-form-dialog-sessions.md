# ADR 202606.003: Multi-form dialog sessions

Status: Accepted

Date: 2026-06-20

## Context

Corusco form models represent committable editing state. The current Swing
dialog controller can turn one `FormModel<R>` into OK, Apply, Cancel,
active-editor commit, dirty-cancel confirmation, validation-summary, baseline
acceptance, reset, and final-result behavior.

Real dialogs often contain multiple independently owned forms. A common example
is a tabbed or sectioned dialog where each tab edits a different concern of the
same logical operation:

- main customer fields;
- invoice settings;
- contact persons;
- security or certificate settings;
- advanced options.

Flattening every field into one generated form would make these screens harder
to own, test, reuse, and migrate incrementally. Each tab or section may have a
natural form model of its own, but the dialog still needs one aggregate dirty
state, validation state, reset operation, baseline acceptance operation, and
final committed result.

ADR 202606.001 allows generated forms to come from records or abstract classes.
ADR 202606.002 introduces generated presentation models and auxiliary
`ComponentStateModel` members for controls, sections, tabs, and dialog button
rows. This decision should build on those layers rather than creating a second
tab-state or presentation-state mechanism.

The generated companion pattern now has two relevant halves. Generated
`*FormModel` classes own semantic editing state and committed results for one
form source. Generated `*PresentationModel` classes own component-state models,
dependency behavior, and binding-facing state for that same form source. A
multi-form dialog needs both halves for each child form, but it also needs a
small aggregate owner that decides how those children form one transaction.
That aggregate owner is not automatically the same kind of generated companion
as a child form.

ADR 202606.002 also separated core form generation from Swing-specific
companion generation. Swing view contracts, behavior plans, and binding classes
are generated only for packages that explicitly request them with
`@SwingCompanionPackage`. Multi-form handling must preserve that boundary: the
core aggregate form session must work when no Swing companions exist, while the
Swing module can provide optional adapters and examples for projects that opt
into Swing companion generation.

The current `FormModel<R>` contract is intentionally small: `problems()`,
`isCommittable()`, `reset()`, `acceptCurrentValues()`, and `toResult()`.
Dirty-state values live on fields, while Swing `FormDialog` accepts a separate
`DirtyState` hook because applications may include nested editors, table
changes, or presenter state in dirty-cancel decisions. A multi-form design must
respect that split instead of adding implicit dirty behavior to every
`FormModel`.

This decision can use generated child form models, but it does not require
annotation-driven parent form generation as the first step.

## Decision

Introduce first-class support for treating multiple child form models as one
committable unit.

The core contract should remain UI-independent. The public concept is a
composite form model: it implements `FormModel<R>`, owns an ordered list of
child forms, delegates reset and baseline acceptance to those children, merges
their problems deterministically, and creates the aggregate result through an
explicit parent assembler.

The likely core API shape is:

```java
public interface CompositeFormModel<R> extends FormModel<R> {
    List<? extends FormModel<?>> children();
}
```

The exact name may change during implementation, but the design requirement is
stable: Corusco must let a dialog treat multiple generated or handwritten form
models as one committable unit without flattening all fields into a single
generated form.

The first implementation should provide a small handwritten composition base in
`corusco-core`, for example:

```java
public abstract class AbstractCompositeFormModel<R>
        implements CompositeFormModel<R> {

    protected abstract R createResult();
}
```

The base should aggregate standard `FormModel` behavior:

- `problems()` returns child problems in child registration order, followed by
  optional parent-level validation problems;
- `isCommittable()` is `false` when any child or parent-level problem has error
  severity;
- `reset()` resets all children and then any parent-owned non-child state, if
  the base class supports such state;
- `acceptCurrentValues()` accepts the baseline on all children and then on
  parent-owned non-child state;
- `toResult()` checks aggregate committability before calling the parent
  assembler.

Parent-level validation is necessary because some rules cross child boundaries.
Examples include "certificate settings are required only when the security tab
selects certificate authentication" or "invoice email must be present when the
billing tab enables electronic invoicing." The parent form should express those
rules as parent problems rather than forcing child forms to know dialog layout
or sibling models.

The parent form owns result assembly. Child forms provide typed partial
results, but they should not know the dialog, tabs, buttons, or final aggregate
type:

```text
CustomerDialogSession
    child: CustomerIdentityFormModel -> CustomerIdentity
    child: BillingSettingsFormModel -> BillingSettings
    child: CertificateSettingsFormModel -> CertificateSettings
    result: CustomerDialogResult
```

In simple cases the parent has one "main" child and several supporting
children. In other cases all children are peers, repeated, or dynamically
selected. The core aggregation contract should not encode tab layout or a
main/sub-form hierarchy. If an application needs a semantic distinction, the
handwritten parent class can expose named accessors such as `identity()`,
`billing()`, and `certificate()`.

Generated form companions should remain directly usable inside handwritten
parent sessions:

```java
public final class CustomerDialogSession
        extends AbstractCompositeFormModel<CustomerDialogResult> {

    private final CustomerIdentityFormModel identity;
    private final BillingSettingsFormModel billing;
    private final CertificateSettingsFormModel certificate;

    @Override
    protected CustomerDialogResult createResult() {
        return new CustomerDialogResult(
                identity.toResult(),
                billing.toResult(),
                certificate.toResult());
    }
}
```

The parent session should normally compose generated child model and
presentation companions, not replace them. A multi-form presenter can keep one
semantic aggregate plus one presentation aggregate:

```text
CustomerDialogSession
    semantic children:
        CustomerIdentityFormModel
        BillingSettingsFormModel
        CertificateSettingsFormModel
    presentation children:
        CustomerIdentityPresentationModel
        BillingSettingsPresentationModel
        CertificateSettingsPresentationModel
    optional dialog state:
        dialogButtons: ComponentStateModel
        applyAction: ComponentStateModel
        revertAction: ComponentStateModel
        selectedTab/problem routing state
```

The semantic aggregate should not own Swing components and should not commit
presentation state. The presentation aggregate, when useful, should be a
handwritten presenter-side object or a later generated companion that groups
child presentation models, dialog button state, tab state, and validation-focus
routing. Keeping it separate avoids making every multi-form transaction pay for
a presentation aggregate when a simple dialog only needs child view bindings
and a dirty-state helper.

That presentation aggregate must also remain toolkit-neutral unless the package
explicitly opts into Swing companions. It may group `ComponentStateModel`
instances and child `*PresentationModel` objects in core/application code. Swing
view contracts, behavior plans, and `*Bindings` should be consumed only by the
Swing presenter layer when they were generated through `@SwingCompanionPackage`.
Applications that bind child views manually should still be able to use the
same composite semantic parent.

The Swing layer should not need a parallel dialog controller for the first
implementation. `FormDialog<P extends FormModel<R>, R>` already accepts any
`FormModel<R>`, so a composite parent form can be passed to the existing
controller. The Swing addition should instead provide helpers that are missing
for multi-form sessions:

- a dirty-state composer that checks field or child dirty state explicitly;
- validation-summary/focus resolver helpers that can map child problems to
  child views or tabs;
- lifecycle examples that register bindings for all child views under one
  `FormDialogLifecycle`.

Dialog actions should have presentation models just like fields, sections, and
tabs. A presenter should be able to expose `ComponentStateModel` instances for
OK, Apply, Cancel, Revert, and button groups without making those actions part
of the committed result. In particular, Apply and Revert need independent
enabled-state rules:

- Apply is relevant only for Apply-enabled dialogs and should be enabled when
  the aggregate form is committable and contains unapplied changes relative to
  the current baseline.
- Revert is relevant only when a Revert policy is configured and should be
  enabled when the transaction differs from the pre-dialog snapshot or when an
  applied value exists that can still be compensated.
- OK and Cancel remain available according to normal dialog policy, although
  their component state may still reflect busy, permission, or validation
  constraints.

These action models belong to the presentation aggregate, not to the semantic
`FormModel`. The semantic form can expose enough state for the presenter to
derive action enablement, but button visibility, enabled state, busy state, and
problem summaries remain presentation concerns.

Dirty-state composition should remain explicit. The core composite contract
does not add `isDirty()` to `FormModel`, because dirty-cancel policy is broader
than form values and existing dialog code deliberately keeps it as a Swing
policy hook. Generated forms may later expose generated dirty-state helpers,
but the initial implementation should use explicit composition:

```java
DirtyState dirty = DirtyStates.any(identityDirty, billingDirty, certificateDirty);
FormDialog<CustomerDialogSession, CustomerDialogResult> dialog =
        new FormDialog<>(session, root, dirty, confirmation);
```

Generated parent/sub-form source annotations are deferred. They should follow
only after handwritten composition proves the API. A future processor stage may
generate a parent session from explicit source declarations, but it must use
typed generated companions and validated member references, not reflection or
handwritten string ids.

## Companion Model Relevance

The companion and presentation model pattern is highly relevant for the current
ADR, but at child-form granularity first.

Use generated companions for:

- child form models, field keys, descriptors, options, dependencies, validators,
  and metadata;
- child presentation models that own field, section, tab, and auxiliary
  component state declared by the child form source;
- generated Swing view contracts, behavior plans, and bindings only when the
  child package opts into them with `@SwingCompanionPackage`;
- typed inventories used by the parent presenter, such as child field keys and
  generated dependency metadata.

Use handwritten composition for the first aggregate session:

- result assembly across children;
- cross-child validation that depends on more than one form;
- aggregate dirty-state policy, including non-form presenter state;
- tab selection or problem-routing policy;
- dialog action-state policy for Apply, Revert, OK, Cancel, and button groups;
- lifecycle ownership of child bindings, whether handwritten or generated, and
  task services.

This split follows the current best-practice boundary in Corusco. Generated
companions remove repetitive, typed, locally derivable wiring. Handwritten
aggregate sessions remain better for domain-specific transaction boundaries,
cross-child rules, and workflow policy that would otherwise require a large
new source annotation language.

The aggregate session must not depend on Swing-specific generated classes. A
core or non-Swing application should be able to compose generated child
`*FormModel` and `*PresentationModel` classes without generating `*View`,
`*BehaviorPlan`, or `*Bindings`. Swing-specific helpers should therefore be
convenience adapters layered over the core aggregate, not required pieces of
the form-handling contract.

The first implementation should therefore avoid a generated
`*DialogSessionModel` unless the examples prove that the handwritten aggregate
has become boilerplate. If generation is added later, it should generate only
from explicit source declarations and should preserve the same two-layer shape:
a semantic aggregate `FormModel<R>` plus an optional presentation aggregate that
groups child `*PresentationModel` instances and dialog-level state.

Efficiency follows from lazy composition. Child presentation models should be
created only for child views that need component state or generated/manual
bindings.
A composite semantic form should not instantiate tab widgets, Swing bindings,
or presentation aggregates. Dynamic or rarely opened tabs can create their
child model/presentation pair when the tab is added, then register that child
with the aggregate through an explicit mutation API only if dynamic children are
in scope for that implementation stage.

## Composition Semantics

Child order is part of the contract. It controls problem aggregation order,
reset order, baseline-acceptance order, and the default order used by validation
summaries. The recommended order is the same order a user encounters the child
forms in the dialog.

Child ownership must be unambiguous. A child form should be registered once in
one composite parent. Sharing the same mutable child instance between two
parents or registering a child twice in one parent should be rejected or clearly
documented as unsupported.

Parent validation runs after child problem collection. Parent validation may
inspect child semantic values, child parse state, or child `isCommittable()`
where needed, but it must not call `child.toResult()` unless the child is
committable. The parent assembler should be the only place that materializes
child committed values for the final result.

Aggregate `toResult()` must fail atomically. If any child or parent-level rule
is not committable, the parent must throw `UncommittableFormException` and must
not partially accept baselines, partially reset children, or partially update a
dialog result.

Apply semantics remain the `FormDialog` semantics. A successful Apply calls
`toResult()` on the composite parent, then accepts current values as the new
baseline on the parent. That baseline acceptance must flow to every child so a
later Cancel reverts only changes made after the last successful Apply.

Cancel semantics remain the `FormDialog` semantics. A confirmed Cancel resets
the composite parent. Reset must flow to all children and should leave the
parent form in the same state it had at its current baseline.

## Apply and Cancel Semantics

The dialog result must distinguish "nothing was ever committed" from "changes
were committed by Apply, then the dialog was closed without further OK."

For a dialog that has never applied successfully, Cancel keeps the familiar
meaning: it closes the dialog with a cancelled result and resets the form to
the original baseline.

For a dialog that has applied successfully at least once, Cancel should be
equivalent to OK at the time of the last successful Apply. The terminal dialog
result should therefore be accepted with the last applied value. Cancel after
Apply does not roll back that applied value; it only discards edits made after
the last successful Apply by resetting the form to the applied baseline before
closing.

The useful cases are:

```text
OK
    -> commit current values, accept current baseline, close accepted(current)

Cancel
    -> reset to original baseline, close cancelled

Apply, OK
    -> Apply commits and accepts an intermediate baseline
    -> OK commits current values, close accepted(current)

Apply, Cancel
    -> Apply commits and accepts an intermediate baseline
    -> Cancel discards only later unapplied edits, close accepted(lastApplied)

Apply, Revert
    -> Apply commits and accepts an intermediate baseline
    -> Revert restores the pre-dialog snapshot, close reverted/cancelled
```

If Apply succeeds more than once, the last successful Apply wins. A failed
Apply must not update the last applied value. If the user changes values after
Apply and then cancels, dirty-cancel confirmation should ask only about those
post-Apply unapplied edits; confirming cancellation preserves the last applied
result and discards the later dirty state.

Revert is a separate optional action for dialogs that want to discard all
changes made since the dialog opened, including values that were already
committed by Apply. This cannot be implemented by ordinary `FormModel#reset()`
after Apply, because Apply intentionally accepts the applied values as the new
baseline. A Revert-capable dialog therefore needs a preserved opening snapshot
or an explicit application-provided revert handler.

Revert should be offered only when the application can make the operation true.
For in-memory or staged edits, that may mean restoring each child form from the
pre-dialog values and clearing the last applied result. For Apply operations
that already wrote to durable storage or called external services, Revert needs
a compensating save/rollback operation and can fail. Corusco should model that
as an optional transaction policy rather than pretending every form model can
reconstruct pre-dialog state from its current baseline.

When Revert succeeds, the terminal result should not be confused with
Apply-Cancel. It should report either a distinct reverted result, if the dialog
result model grows such a state, or a cancelled result with documented
pre-dialog restoration. The important invariant is that `Apply, Cancel`
preserves the last applied value, while `Apply, Revert` restores the opening
state.

Apply and Revert enablement must use different dirty concepts:

- current-baseline dirty means the user has changes that have not yet been
  applied or accepted;
- pre-dialog dirty means the dialog transaction has diverged from the values it
  had when it opened, including successful Apply operations.

Apply should usually follow current-baseline dirty state. It should become
disabled immediately after a successful Apply because there are no unapplied
changes left. Revert should follow pre-dialog dirty state. It may remain enabled
after Apply because the dialog transaction still differs from the opening
snapshot. This gives users both efficient Apply behavior and an explicit way to
undo the whole dialog transaction when the application supports that operation.

Presentation state remains outside the semantic composite form. Generated
`ComponentStateModel` members should continue to own enabled, visible,
editable, relevant, busy, and problem-summary state for fields, sections, tabs,
action buttons, and button rows. Multi-form sessions may coordinate those
presentation models, but they should not commit them into aggregate results.

## Consequences

Tabbed and multi-section dialogs can coordinate commit, Apply, Cancel, reset,
and validation through a first-class model rather than application-specific
glue.

Generated child forms stay small and reusable. A tab can have its own generated
source, generated model, generated presentation model, optional Swing bindings,
and tests.

Parent sessions become the natural place for cross-child validation and final
result assembly. This keeps child forms independent of dialog layout and keeps
the dialog controller independent of domain-specific result construction.

The design preserves the existing `FormDialog` API. A composite parent is still
just a `FormModel<R>`, so single-form dialogs and multi-form dialogs share the
same accept/apply/cancel implementation.

The design intentionally leaves dirty-state composition explicit. That avoids a
breaking change to `FormModel`, but it means the first implementation must
provide good helper APIs and examples so users do not rebuild fragile dirty
logic by hand.

The design increases the importance of problem targeting. Aggregated problems
must remain useful for validation summaries and focus navigation across child
views. Existing `ProblemTarget.field(...)` is enough for many cases, but
examples should show how a resolver routes a child problem to the right
component or tab.

## Critical Review

The first draft risked overloading "session" to mean both semantic aggregate
model and Swing dialog lifecycle. The implementation should keep the semantic
composite in `corusco-core` and keep modal lifecycle, focus, keyboard, and
dirty-cancel policy in `corusco-swing`.

The first draft suggested extending `FormModel` with children but did not
resolve dirty state. That would be misleading because `FormModel` does not
currently expose dirty state. Dirty-state helpers belong beside `DirtyState`
unless a later ADR adds dirty-state read APIs to form models.

The first draft used a main/sub-form hierarchy as if it were the core model.
That is too narrow for peer tabs, repeated child sections, and selected-row
detail forms. Named children can be exposed by parent classes, but the base
contract should be ordered composition.

The first draft did not specify whether parent-level validation exists. Without
it, cross-tab rules would either be duplicated in children or moved into Swing
presenters. Parent-level validation must be part of the core composite model.

The first draft did not define ordering and ownership rules. Without those
rules, problem summaries and reset/baseline behavior can become nondeterministic
or double-apply to the same mutable child.

The first draft did not guard against a parallel dialog controller. A second
controller would duplicate `FormDialog` behavior and increase regression risk.
The first implementation should prove that the existing controller works with a
composite parent before adding new controller APIs.

The first draft did not separate presentation-region state from semantic
commit. ADR 202606.002 already gives Corusco a place for tab, section, and
button state. Multi-form sessions should coordinate those models, not replace
them.

The first draft also underplayed the companion model introduced by the previous
ADR. Ignoring child `*PresentationModel` companions would push tab state,
dependency state, and generated binding state back into handwritten Swing code.
Overcorrecting in the other direction would generate aggregate session classes
before the handwritten API has proven its shape. The efficient middle is to
compose generated child companions now and defer generated aggregate companions
until repeated examples expose real boilerplate.

The first draft also risked assuming that generated Swing bindings always
exist. That would conflict with ADR 202606.002's opt-in Swing companion model.
The aggregate form API belongs in core and must consume only core generated
types. Swing-generated `*View`, `*BehaviorPlan`, and `*Bindings` classes are
optional accelerators for Swing presenters, not prerequisites for multi-form
sessions.

The current single-form dialog implementation already accepts the current
baseline on Apply, but its terminal `result()` historically remained cancelled
until OK. This ADR treats that as insufficient for Apply-enabled dialogs. Once
Apply has successfully committed a value, a later user Cancel should report the
last applied value as accepted; otherwise callers cannot distinguish "user
discarded the whole operation" from "user applied changes and then closed the
dialog."

The current form baseline API also does not preserve the pre-dialog baseline
after Apply. That is fine for OK, Cancel, and Apply-Cancel, but it means Revert
must be designed as an explicit optional transaction feature. Adding Revert by
quietly changing `reset()` to mean "original opening value" would break the
existing Apply baseline semantics and make long-lived editors less predictable.

Apply/Revert button enablement is another reason to keep two dirty concepts.
Reusing only the current form baseline would incorrectly disable Revert after
Apply, while reusing only the pre-dialog baseline would keep Apply enabled even
after changes have already been applied. The dialog presentation model should
therefore derive separate action states from separate dirty sources.

## Out of Scope

This decision does not introduce annotation-driven parent/sub-form generation.
The first implementation should be handwritten composition plus examples.

This decision does not add dirty state to `FormModel`. Dirty-cancel policy
continues to use `DirtyState` and explicit composition.

This decision does not require every dialog to offer Revert. Revert is optional
and only valid when the caller can restore or compensate all applied and
unapplied changes back to the pre-dialog state.

This decision does not define visual tab layout, wizard navigation, generated
layout DSLs, or native `JDialog` ownership. Those remain presentation concerns.

This decision does not make Swing companion generation mandatory. Projects may
use composite form sessions with generated core form/presentation models,
handwritten Swing views, other UI toolkits, or tests that do not generate Swing
companions.

This decision does not solve editable table rows, master/detail collection
editing, or repeated sub-form identity. A composite parent should not prevent
those later designs, but the first implementation should target fixed child
forms.

This decision does not change inactive-value semantics from ADR 202606.002.
Dependency-driven hidden, disabled, cleared, retained, or ignored values remain
future editor/dependency work.

## Implementation Plan

1. Add `CompositeFormModel<R>` or the chosen equivalent to
   `corusco-core/src/main/java/cz/auderis/corusco/core/form`.
2. Add an `AbstractCompositeFormModel<R>` base that stores ordered child forms,
   rejects null children, exposes an immutable child list, aggregates problems,
   delegates reset and baseline acceptance, checks aggregate committability, and
   calls a protected result assembler.
3. Add a parent-validation hook to the base, mirroring
   `AbstractFormModel#validationProblems()`.
4. Add core tests for child problem ordering, parent-level validation,
   committability, reset order, baseline acceptance order, null/duplicate child
   handling, and uncommittable `toResult()` behavior.
5. Add Swing dirty-state helpers near `DirtyState`, such as `DirtyStates.any`,
   while keeping them independent of `FormModel`.
6. Add Swing tests proving that existing `FormDialog` OK, Apply, Cancel,
   dirty-cancel confirmation, and validation-summary behavior works when the
   dialog model is a composite parent.
7. Add dialog tests for terminal-result semantics: Cancel before Apply is
   cancelled, Apply followed by Cancel is accepted with the last applied value,
   Apply followed by further edits and confirmed Cancel discards only the later
   edits, and failed Apply does not update the last applied result.
8. Define an optional Revert policy separately from `FormModel#reset()`,
   including how a dialog preserves an opening snapshot or delegates to a
   compensating application handler, and how the terminal result distinguishes
   Revert from Apply-Cancel.
9. Add dialog action presentation models for Apply and Revert, with enablement
   derived from current-baseline dirty state and pre-dialog dirty state
   respectively.
10. Add example presenter code that composes generated child `*PresentationModel`
   instances with the handwritten semantic parent, while keeping dialog-level
   button and tab state out of committed results and without requiring generated
   Swing bindings.
11. Add a `ProblemFocusResolver` example or helper that routes child field
   problems to child components and can reveal/select the owning tab before
   requesting focus.
12. Add an example multi-form dialog session in `corusco-examples`, preferably a
   tabbed customer/security/settings scenario using generated child form models
   and presentation models with a handwritten parent session. If the example
   also uses generated Swing bindings, it must do so through an explicit
   `@SwingCompanionPackage` adapter package and keep the core session usable
   without those bindings.
13. Update package docs for `corusco-core` form models and `corusco-swing`
   dialogs to mention composite form models without making them the default
   path for simple dialogs.
14. Validate with the relevant focused tests, full `test`, generated Javadocs,
    `git diff --check`, and full `build` before considering the ADR ready to
    accept.

## Acceptance Criteria

The implementation is acceptable when a generated-child, handwritten-parent
multi-form dialog can:

- report child and parent validation problems through one `FormModel`;
- block OK and Apply when any child or parent rule has an error;
- apply successfully and update every child baseline;
- return the last successful applied result as accepted when the user cancels
  after Apply;
- discard only post-Apply unapplied edits during Apply-Cancel;
- optionally revert all applied and unapplied changes to the pre-dialog state
  when an explicit Revert policy is configured;
- expose Apply and Revert component state so Apply is disabled with no
  current-baseline changes and Revert is disabled with no pre-dialog changes;
- cancel with explicit aggregate dirty-state confirmation;
- reset every child after confirmed cancellation;
- assemble one final typed result without children knowing the dialog layout;
- show a validation summary and focus the first relevant child component or
  tab;
- compose generated child presentation models for tab, section, dependency, and
  component state without committing that state into the result;
- run the core aggregate-session tests without generated Swing companions;
- keep generated child forms and presentation models usable outside the parent
  session.
