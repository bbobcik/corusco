# ADR 202606.003: Multi-form dialog sessions

Status: Proposed

Date: 2026-06-19

## Context

Corusco form models represent committable editing state. Dialog support can
turn one form model into OK, Apply, Cancel, dirty-state, and validation-summary
behavior.

Real dialogs often contain multiple forms. A common example is a tabbed dialog
where each tab edits a different concern of the same logical operation:

- main customer fields;
- invoice settings;
- contact persons;
- security or certificate settings;
- advanced options.

Flattening every field into one generated form would make these screens harder
to own, test, and reuse. Each tab or section may have a natural form model of
its own, but the dialog still needs one aggregate dirty state, validation
state, reset action, baseline acceptance operation, and final result.

This decision can use generated forms from records or abstract classes, but it
does not require annotation-driven parent form generation as the first step.

## Decision

Support dialogs that contain more than one form model by introducing a parent
form session containing one main form and zero or more sub-forms. The parent
coordinates dialog-level behavior: aggregate dirty state, aggregate validation
problems, OK/Apply enablement, reset, baseline acceptance, and final result
assembly.

The main/sub-form distinction is useful when one result owns the operation and
other forms supply nested values:

```text
CustomerDialogSession
    main form: CustomerEditFormModel
    sub-form: BillingSettingsFormModel
    sub-form: CertificateSettingsFormModel
```

The parent-child model is useful when sub-forms can be repeated or dynamically
created, such as one sub-form per tab, row, or selected entity. Both cases
should share the same aggregation contract.

Introduce a core aggregation abstraction before adding dialog-specific Swing
behavior:

```java
public interface FormGroup<R> extends FormModel<R> {
    List<FormModel<?>> children();
}
```

The exact API may differ, but the design requirement is stable: Corusco must
let a dialog treat multiple form models as one committable unit without
flattening every field into a single generated form. Generated form companions
should remain usable inside handwritten parent sessions.

Generated support should be conservative. The first implementation can provide
handwritten `FormGroup` composition in `corusco-core` and dialog aggregation in
`corusco-swing`. Annotation-driven parent/sub-form generation can follow after
the base composition contract is proven.

## Consequences

Tabbed and multi-section dialogs can coordinate dirty state and validation
through a first-class model rather than application-specific glue.

Generated form companions remain useful inside handwritten parent sessions.

Multi-form dialog aggregation needs clear ownership of result creation. A
parent session should assemble results; child forms should not know dialog
layout or tab structure.

The aggregation contract must define how parse problems, validation problems,
dirty state, reset, and baseline acceptance compose across children.

## Implementation Notes

1. Add a core `FormGroup` or equivalent aggregation contract.
2. Implement aggregate problem, dirty, reset, baseline, and committability
   behavior.
3. Wire dialog dirty/problem aggregation through the group contract.
4. Add a tabbed-dialog example using generated child form models.
5. Consider annotation-driven parent/sub-form generation only after handwritten
   composition is proven.
