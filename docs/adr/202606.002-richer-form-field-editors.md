# ADR 202606.002: Richer generated form field editors

Status: Accepted

Date: 2026-06-20

## Context

Generated form fields currently cover text fields, date text fields, combo
boxes, and boolean checkboxes. This is enough for the first generated-form
pipeline but not enough for practical business Swing forms.

Applications commonly need radio selectors, custom checkbox mappings,
checkbox groups, numeric sliders or spinners, and dialog-selected values such
as colors, fonts, files, directories, and certificates. Some of these values
can be summarized as text, but text is not their authoritative edit model.

The business-control space is broader than a simple list of Swing classes.
Common application forms also need typed identifiers selected from reference
data, multi-selection option sets, nullable or unknown states, masked or
protected values, bounded numeric input, duration or time input, path-like
values, icon or image choices, and security material selected through dedicated
dialogs. The generated metadata should therefore describe editor families and
value semantics rather than try to enumerate every concrete Swing component in
the first implementation.

Corusco is still before its first official release, so this decision may reshape
the field metadata API instead of preserving a too-small first draft. The goal
is an elegant generated-form model for advanced business applications, not a
minimal extension that keeps early internal metadata intact.

This decision is separate from mixed generated form sources. It applies to both
record-based forms and abstract-class forms once abstract-class support exists.

## Business Control Examples

The richer editor model should be pressure-tested against common business
application controls and workflows. These examples do not all need built-in
Swing widgets in the first implementation, but the generated metadata and
binding extension surface should be able to describe them without bypassing the
form model:

- reference-data lookup fields, such as customer, account, project, cost
  center, tax code, or country selectors, where the semantic value is a typed
  id or reference object and the editor may be a combo box, autocomplete field,
  search dialog, or recent-value picker;
- dependent picklists, such as country/state, organization/team/user, product
  category/product, or document type/status, where allowable values and
  relevance depend on other fields;
- multi-selection controls, such as role assignment, notification channels,
  feature flags, approver groups, tags, or labels, represented by enum sets or
  typed collection values rather than arrays of checkbox state;
- tri-state and mapped boolean controls, such as enabled/disabled/inherited,
  yes/no/unknown, approved/rejected/pending, or send/do-not-send/default,
  where the checkbox state is not itself the semantic value;
- bounded numeric business inputs, such as priority, SLA days, retry count,
  percentage, discount, quantity, tolerance, or threshold, with range, step,
  unit, and formatting metadata;
- monetary and measured values, such as amount plus currency, quantity plus
  unit, or rate plus period, where the editor may need multiple coordinated
  controls but the form field should still expose one semantic value;
- date/time variants, such as date range, effective-from/to, recurring window,
  duration, timezone-aware timestamp, or business-calendar date, which need more
  metadata than a plain date text field;
- protected or sensitive values, such as password, token, API key, secret
  reference, or certificate, where masking, reveal permission, reset state, and
  validation feedback are presentation state layered over a semantic value;
- attachment and external-resource pickers, such as file, directory, document
  template, image, icon, certificate, keystore alias, or URL, where a compact
  summary is displayed but a richer picker edits the actual value;
- workflow state controls, such as sections, tabs, wizard pages, button rows,
  approval panels, or validation summaries, which are not semantic form values
  but need generated component-state models for visibility, enabled state,
  busy state, and problem summaries.

SwingX and JGoodies provide useful prior art for deciding which problems belong
in generated metadata rather than in one-off Swing wiring:

- SwingX-style date pickers, search fields, login panes, task panes, status
  bars, tables, and tree tables show that business applications need richer
  control families than plain text fields and combo boxes. Corusco should not
  copy those component APIs, but should capture their recurring metadata:
  picker intent, search/filter semantics, protected credential state, section
  state, status/problem reporting, and row selection summaries.
- JGoodies Binding points toward generated presentation models that expose
  value models, list selection models, and component state independently of
  concrete Swing components. Corusco should generate typed field models,
  option sources, selected-value state, and auxiliary component-state members
  that can be bound to standard Swing, SwingX-like widgets, or application
  widgets.
- JGoodies Validation points toward treating validation and problem reporting
  as first-class model state. Corusco field dependencies, option descriptors,
  and component-state models should expose validity, warning/error summaries,
  and inactive-validation behavior rather than leaving every panel to invent
  its own error labels.
- JGoodies Forms is layout-oriented and therefore mostly outside this ADR, but
  its emphasis on consistent form construction is a reminder that generated
  view contracts need stable labels, gaps, grouped sections, and repeated
  action rows. Those concerns should be enabled through resource keys and
  component-state models even though the ADR does not define a layout DSL.

The table and tree-table cases are deliberately treated as boundary examples.
A single generated form field may reference a selected row, selected id, or
collection value, but editable grids, master/detail screens, and sub-form
collections belong to the table/sub-form design rather than to this editor
family decision. This ADR should still make sure generated keys, option
descriptors, validation summaries, and component-state models are reusable by
those later designs.

## Decision

Separate field value semantics from editor family. The core model remains
based on semantic `FieldModel<O, T>` and text-preserving
`TextFieldModel<O, T>`. An annotation chooses the intended editor family and
metadata; Swing bindings decide how a specific component edits the semantic
value.

Replace the small fixed `FieldKind` enum with a richer editor-family descriptor.
The descriptor should carry the information that does not fit in an enum:
option source, selection cardinality, nullability policy, value mapping, range
metadata, picker type, summary renderer contract, and binding expectations.
Existing text, date, combo-box, and checkbox fields become simple cases of this
descriptor model.

The editor-family model should distinguish these concepts:

- semantic value type: the Java value stored in `FieldModel` or
  `TextFieldModel`;
- editor family: text, date text, combo box, radio group, checkbox, checkbox
  group, numeric range, picker, or rich opaque value;
- option source: enum constants, static generated options, application-provided
  options, or dynamically observed options;
- selection shape: single value, nullable single value, boolean two-state,
  three-state, set-like multi-selection, or bounded numeric value;
- display summary: the text or icon shown when the real editor is a dialog or
  richer component;
- binding contract: the Swing component family and behavior plan that can edit
  the semantic value.

The first implementation plan should be ambitious enough to prove the model
across simple, mapped, grouped, ranged, and picker-based editors. Add editor
families in this priority order:

1. Enum radio group as a single-selection alternative to enum combo boxes.
2. Numeric range editors for integer and decimal values, starting with spinner
   or slider metadata over existing semantic numeric models.
3. Checkbox value mapping for non-boolean two-state and nullable three-state
   business values.
4. Checkbox groups for enum-backed `EnumSet<E>` values, with custom set-like
   values deferred until the generated option-source model is stable.
5. Picker fields for color, font, file, directory, and similar dialog-selected
   values.
6. Opaque rich values such as certificate-like selections where compact text is
   only a summary of the selected value.

The priority is intentionally based on generated-form leverage and model
coverage. Radio groups and numeric ranges exercise option/range metadata.
Checkbox mappings and groups prove richer value mapping. Picker and opaque
values prove the separation between semantic value, display summary, chooser
action, and dialog ownership.

Picker-style fields must bind to semantic values. A text field or label may
display a summary, but the summary is not the authoritative form value. For
example, a file picker may display a path string, but the field value should be
a `Path` or application-specific selection object according to the source
member type. A certificate picker may display subject and expiry text, but the
field value remains the certificate or certificate reference.

The processor should continue generating typed companions rather than requiring
users to write string identifiers by hand. New annotations or annotation
attributes should describe editor intent at the source declaration, and the
processor should normalize them into the generated descriptor, field-key,
form-model, view-contract, resource, validation, and behavior-plan output.
The processor must also treat diagnostics as part of the feature contract:
duplicate option keys, unresolved dependency references, unsupported semantic
types, contradictory editor annotations, invalid range bounds, and resource-key
collisions should fail at compile time.

Advanced business form use cases should influence the core design from the
start. A new editor family is a good candidate when it has stable
cross-application semantics, can be represented by typed generated metadata,
can be tested without application-specific Swing wiring, and can preserve the
semantic-value model. Highly product-specific controls should still be able to
consume generated keys, descriptors, resources, and validation metadata through
an extension point rather than by bypassing Corusco's generated companions.

## Described and Allowable Values

Corusco should adopt the useful part of NiFi's `AllowableValue` pattern, but
keep it typed and aligned with Corusco's resource model. Option-like editors
need more than the semantic Java value:

- a stable value used by the generated field model;
- a generated option key or id for testing, persistence, and diagnostics;
- generated resource keys for labels, descriptions, and help;
- optional icon or visual hint metadata from the descriptor layer;
- optional disabled or discouraged reason from the descriptor layer or dynamic
  option provider;
- optional grouping or ordering metadata.

This suggests a typed option descriptor, for example
`OptionDescriptor<T>`, rather than using strings as the common denominator.
Enum constants can generate these descriptors directly. Application-provided
options can expose the same descriptor shape. Combo boxes, radio groups,
checkbox groups, segmented controls, and custom option editors should consume
one option model instead of inventing per-editor representations.

The value-side contract should be deliberately narrow. A domain enum or value
may provide a stable option key, but it should not provide labels,
descriptions, help text, disabled reasons, or other presentation metadata that
belongs in resources and generated descriptors. If an interface is needed, it
should be named for identity rather than description, for example
`OptionValueIdentity`, with no display methods.

Annotation names introduced by this ADR should be Corusco/form-specific rather
than common nouns. The narrowly form-owned annotations therefore live under the
`@CoruscoForm` namespace: `@CoruscoForm.Option`,
`@CoruscoForm.ComponentState`, `@CoruscoForm.DependsOn`, and
`CoruscoForm.DependencyEffect`. This keeps the public source shape close to the
form contract while avoiding fragile imports named only `Option`,
`ComponentState`, or `DependsOn`. Broader editor-family annotations such as
`@RadioGroup` can remain top-level because they are concrete Corusco form
editors rather than generic metadata words.

The allowable-value set should also be validation metadata. If a generated
field declares a closed option set, committing a value outside that set is a
validation problem unless the editor family explicitly allows custom values.
This keeps the UI option list, generated descriptor, and validation contract in
one place.

An enum can provide stable identity and ordering directly:

```java
public enum AuthenticationMode {
    @CoruscoForm.Option(key = "password", order = 1)
    PASSWORD,

    @CoruscoForm.Option(key = "certificate", order = 2)
    CERTIFICATE,

    @CoruscoForm.Option(key = "external", order = 3)
    EXTERNAL
}
```

The processor can turn that identity into typed generated option descriptors
that point at resources. Fully-qualified resource keys are derived, not
handwritten. The common prefix comes from the form resource namespace and the
field member name. In the examples below, `@CoruscoForm(id = "customer/security")`
provides the form namespace, `authenticationMode` provides the field segment,
and `@CoruscoForm.Option(key = "...")` provides the option segment:

```text
customer/security.authenticationMode.password.label
|---------------| |----------------| |------| |---|
form namespace    field member       option   facet
```

The first version should use the `@CoruscoForm` id as the form resource namespace
unless an explicit resource-prefix annotation is introduced. That keeps one
visible source for the common prefix and avoids duplicating resource strings on
every option.

```java
public final class CustomerSecurityEditOptions {

    private static final OptionResourcePrefix AUTHENTICATION_MODE_RESOURCES =
            OptionResourcePrefix.of(
                    CustomerSecurityEditForm.RESOURCE_NAMESPACE,
                    CustomerSecurityEditFields.AUTHENTICATION_MODE);
    private static final OptionKey PASSWORD_KEY = OptionKey.of("password");
    private static final OptionKey CERTIFICATE_KEY = OptionKey.of("certificate");
    private static final OptionKey EXTERNAL_KEY = OptionKey.of("external");

    public static final OptionDescriptor<AuthenticationMode> PASSWORD =
            OptionDescriptor.of(
                    AuthenticationMode.PASSWORD,
                    PASSWORD_KEY,
                    AUTHENTICATION_MODE_RESOURCES.label(PASSWORD_KEY),
                    AUTHENTICATION_MODE_RESOURCES.description(PASSWORD_KEY),
                    AUTHENTICATION_MODE_RESOURCES.help(PASSWORD_KEY));
    public static final OptionDescriptor<AuthenticationMode> CERTIFICATE =
            OptionDescriptor.of(
                    AuthenticationMode.CERTIFICATE,
                    CERTIFICATE_KEY,
                    AUTHENTICATION_MODE_RESOURCES.label(CERTIFICATE_KEY),
                    AUTHENTICATION_MODE_RESOURCES.description(CERTIFICATE_KEY),
                    AUTHENTICATION_MODE_RESOURCES.help(CERTIFICATE_KEY));
    public static final OptionDescriptor<AuthenticationMode> EXTERNAL =
            OptionDescriptor.of(
                    AuthenticationMode.EXTERNAL,
                    EXTERNAL_KEY,
                    AUTHENTICATION_MODE_RESOURCES.label(EXTERNAL_KEY),
                    AUTHENTICATION_MODE_RESOURCES.description(EXTERNAL_KEY),
                    AUTHENTICATION_MODE_RESOURCES.help(EXTERNAL_KEY));

    public static final List<OptionDescriptor<AuthenticationMode>>
            AUTHENTICATION_MODE = List.of(PASSWORD, CERTIFICATE, EXTERNAL);

    }
}
```

Application-provided options should expose the same shape:

```java
OptionSource<CertificateReference> certificates =
        OptionSource.dynamic(
                CustomerSecurityEditFields.CERTIFICATE,
                certificateStore::availableCertificates);
```

Dynamic option sources must have stable option keys for the same semantic
value across refreshes. They also need observable loading and failure state so
bindings can present a busy or problem state without replacing the field value
with transport errors. If the current semantic value disappears from a refreshed
closed option source, validation should report that the value is no longer
allowable unless the editor descriptor explicitly permits stale values.

## Field Dependencies

Corusco should support dependency metadata comparable to NiFi's `dependsOn`,
adapted to typed form fields. A dependent field can be relevant only when one
or more source fields have matching semantic values. When a dependency is not
satisfied, generated behavior can hide or disable the field, skip required-field
validation, and exclude the field from blocking commit.

Dependencies should be expressed in generated metadata, not as ad-hoc Swing
listeners. The first form can be conservative:

- depend on another generated field key;
- match a closed set of typed values or option descriptors;
- combine multiple dependencies with logical `and`;
- define the inactive policy: hidden, disabled, read-only, cleared, or retained;
- define whether inactive fields are ignored by validation and dirty-state
  aggregation.

This dependency model belongs near editor descriptors because it affects both
presentation and validation. It should not become a general workflow engine.
Cross-field business rules that cannot be expressed as simple relevance
dependencies remain ordinary form validation rules.

The source-level form can begin with simple dependency declarations:

```java
@CoruscoForm(id = "customer/security")
public abstract class CustomerSecurityEdit {

    @RadioGroup
    public abstract AuthenticationMode authenticationMode();

    @TextField
    @CoruscoForm.DependsOn(
            field = "authenticationMode",
            values = "PASSWORD",
            effect = CoruscoForm.DependencyEffect.VISIBLE)
    public abstract String password();

    @Picker(kind = PickerKind.CERTIFICATE)
    @CoruscoForm.DependsOn(
            field = "authenticationMode",
            values = "CERTIFICATE",
            effect = CoruscoForm.DependencyEffect.ENABLED)
    public abstract CertificateReference certificate();
}
```

The current vertical slice keeps dependency annotation values as source tokens,
but the processor resolves those tokens at compile time. Enum dependencies
accept enum constant names and generated option keys, checkbox dependencies
accept `true` or `false`, and text/date-text dependencies continue to compare
raw text. Generated metadata should therefore make the rule inspectable without
reading annotations reflectively and without preserving unvalidated value
strings:

```java
public final class CustomerSecurityEditDependencies {

    public static final FieldDependency<?> PASSWORD =
            FieldDependency.of(
                    CustomerSecurityEditFields.AUTHENTICATION_MODE,
                    "passwordState",
                    List.of(AuthenticationMode.PASSWORD),
                    DependencyEffect.VISIBLE);
    public static final FieldDependency<?> CERTIFICATE =
            FieldDependency.of(
                    CustomerSecurityEditFields.AUTHENTICATION_MODE,
                    "certificateState",
                    List.of(AuthenticationMode.CERTIFICATE),
                    DependencyEffect.ENABLED);

    public static final List<FieldDependency<?>> ALL =
            List.of(PASSWORD, CERTIFICATE);

    private CustomerSecurityEditDependencies() {
    }
}
```

Inactive policy needs explicit value and commit semantics. `HIDDEN`,
`DISABLED`, and `READ_ONLY` describe presentation state; `CLEARED` and
`RETAINED` describe what happens to the semantic value while inactive. The
generated form model should keep the raw field value inspectable either way,
but commit and validation must follow the dependency metadata: ignored inactive
fields do not block commit, cleared inactive fields commit the configured empty
value, and retained inactive fields preserve the last semantic value. These
inactive policies and fluent dependency builders are future API work, not part
of the current vertical slice.

## Component State Models

Corusco also needs a counterpart to JGoodies' component-state binding idea, but
as generated form infrastructure rather than as direct Swing model code.
`FieldModel<O, T>` owns the semantic value, dirty state, and touched state.
Advanced forms also need observable presentation state that can be attached to
the actual control: enabled, visible, editable, required-now, relevant, busy,
protected, valid, problem summary, and possibly focus or selection hints.

State models should be generated for a field only when the field declaration
asks for one or when another generated feature requires one. A simple generated
text field that has no dynamic state should not pay for a state model. A field
with a dependency rule, protected-value behavior, busy-state binding, explicit
presenter-controlled state, or custom editor state can opt in through an
annotation or generated metadata.

For an opted-in field, the processor should generate a normal semantic form
model and a separate presentation model that owns component-state models. The
generated binding attaches both semantic and presentation state to the form
control: the form model edits the semantic value, while the presentation model
drives Swing properties and decorations. The state model is not itself a Swing
component and does not replace `FieldModel`.

A form presenter should be able to control generated state models explicitly.
For example, presenter code may disable a field during a background lookup,
mark a control read-only after a permission check, reveal advanced settings, or
set a busy/problem summary on a picker. The same state model may also be
controlled automatically by generated dependency behavior. When both presenter
code and generated rules affect the same state, the API must define
composition rules instead of letting the last listener win accidentally.

The likely composition model is layered state:

- base state from generated descriptors;
- dependency-derived relevance and inactive policy;
- validation-derived problem and valid state;
- presenter overrides for explicit workflow decisions;
- transient operation state such as busy or loading.

The effective component state is what bindings apply to Swing controls. The
raw layer values remain inspectable for tests and diagnostics.

Component state models should also be available without a corresponding form
field. In the current vertical slice, abstract form sources can declare those
auxiliary state members using annotated methods that return
`ComponentStateModel`; the generated presentation model owns the actual state.
These members can represent groups of controls, titled sections, dialog
buttons, wizard pages, tab headers, or entire dialog tabs. A presenter can then
centrally manage a tab's enabled, visible, busy, valid, or problem-summary
state without inventing local Swing listeners. Dependency rules may also target
these auxiliary state members, for example hiding a certificate tab until a
security mode field selects certificate authentication.

Direct source declarations keep the generated API reviewable: methods are
present in source, the processor classifies them as presentation state rather
than committed result values, and generated/runtime code uses typed keys
instead of free-form string ids. Any source-name references that remain in
annotations must be validated against members of the same form source and
normalized to generated typed keys.

Component-state members are not value members. Their observable state must live
on the generated presentation model, not on the generated form model or
committed form result. They must not participate in dirty-value comparison,
equality, generated copy/update methods, result construction, or validation
rules except through explicit state or dependency metadata. This distinction is
especially important for abstract form sources, where both semantic value
members and auxiliary state declarations may appear as methods on the same
source type.

This layer is the right place to combine editor metadata, dependency results,
validation state, auxiliary presentation regions, and explicit presenter
overrides. It gives generated forms a reviewable path for common business UI
state without pushing Swing state into the semantic form model.

Presenter code should interact with generated state models directly:

```java
final class CustomerSecurityPresenter {

    private final CustomerSecurityEditPresentationModel presentation;

    CustomerSecurityPresenter(CustomerSecurityEditPresentationModel presentation) {
        this.presentation = presentation;
    }

    void refreshCertificates() {
        presentation.certificateState().busy().setValue(true, ChangeOrigin.SYSTEM);
        certificateService.refreshAsync()
                .whenComplete((result, failure) -> {
                    presentation.certificateState().busy()
                            .setValue(false, ChangeOrigin.SYSTEM);
                    presentation.certificateState().problemSummary()
                            .setValue(problemFor(failure), ChangeOrigin.SYSTEM);
                });
    }

    void applyPermissions(UserPermissions permissions) {
        presentation.passwordState().editable()
                .setValue(permissions.canEditSecrets(), ChangeOrigin.SYSTEM);
        presentation.advancedSection().visible()
                .setValue(permissions.canSeeAdvancedSecurity(), ChangeOrigin.SYSTEM);
    }
}
```

Generated Swing bindings would attach both value and state models to controls:

```java
CustomerSecurityEditFormModel model =
        new CustomerSecurityEditFormModel(original);
CustomerSecurityEditPresentationModel presentation =
        new CustomerSecurityEditPresentationModel(model);
CustomerSecurityEditView view = new CustomerSecurityEditPanel();

BindingScope bindings = CustomerSecurityEditBindings.bind(view, presentation);
bindings.install();

// Equivalent generated behavior, shown only for shape:
BindingFactory.textField(view.passwordField(), presentation.form().password());
BindingFactory.componentState(view.passwordField(), presentation.passwordState());
BindingFactory.componentState(view.certificateTab(), presentation.certificateTab());
```

Auxiliary state models should allow central control of non-field UI regions:

```java
presentation.certificateTab().enabled()
        .setValue(presentation.form().authenticationMode().value().value()
                == AuthenticationMode.CERTIFICATE, ChangeOrigin.SYSTEM);
presentation.dialogButtons().busy()
        .setValue(saveOperation.running(), ChangeOrigin.SYSTEM);
```

The auxiliary state members are explicit source, not a list of string ids:

```java
@CoruscoForm(id = "customer/security")
public abstract class CustomerSecurityEdit {

    @CoruscoForm.ComponentState
    public abstract ComponentStateModel advancedSection();

    @CoruscoForm.ComponentState
    @CoruscoForm.DependsOn(
            field = "authenticationMode",
            values = "CERTIFICATE",
            inactive = InactivePolicy.HIDDEN)
    public abstract ComponentStateModel certificateTab();

    @CoruscoForm.ComponentState
    public abstract ComponentStateModel dialogButtons();
}
```

## Target Source Shape

An annotated source should stay compact while carrying enough intent for the
processor to generate typed field models, editor descriptors, option metadata,
dependency plans, component-state models, and behavior plans:

```java
@CoruscoForm(id = "customer/security")
public abstract class CustomerSecurityEdit {

    @RadioGroup
    public abstract AuthenticationMode authenticationMode();

    @TextField
    @CoruscoForm.DependsOn(field = "authenticationMode", values = "PASSWORD")
    @CoruscoForm.ComponentState
    @ProtectedValue
    public abstract String password();

    @Picker(kind = PickerKind.CERTIFICATE)
    @CoruscoForm.DependsOn(field = "authenticationMode", values = "CERTIFICATE")
    @CoruscoForm.ComponentState
    public abstract CertificateReference certificate();

    @NumericRange(min = "1", max = "90", step = "1")
    @CoruscoForm.ComponentState
    public abstract int rotationDays();

    @CheckBoxGroup
    // Future option-source annotation shape is still TBD.
    public abstract EnumSet<Permission> permissions();

    @CoruscoForm.ComponentState
    public abstract ComponentStateModel advancedSection();

    @CoruscoForm.ComponentState
    @CoruscoForm.DependsOn(
            field = "authenticationMode",
            values = "CERTIFICATE",
            inactive = InactivePolicy.HIDDEN)
    public abstract ComponentStateModel certificateTab();

    @CoruscoForm.ComponentState
    public abstract ComponentStateModel dialogButtons();
}
```

Generated companions would expose typed handles rather than string ids:

```java
CustomerSecurityEditFormModel model =
        new CustomerSecurityEditFormModel(original);
CustomerSecurityEditPresentationModel presentation =
        new CustomerSecurityEditPresentationModel(model);
GeneratedCustomerSecurityEdit committed =
        model.toResult();

FieldModel<CustomerSecurityEdit, AuthenticationMode> mode =
        model.authenticationMode();
ComponentStateModel passwordState =
        presentation.passwordState();
ComponentStateModel advancedSection =
        presentation.advancedSection();
```

## Consequences

New field editors can share typed keys, descriptors, validation, resources, and
generated behavior plans instead of becoming ad-hoc Swing code.

Generated form metadata becomes a better description of intended presentation
families rather than a small list of Swing component classes.

Replacing `FieldKind` with a proper editor descriptor is a breaking internal
API change, but it prevents the generated metadata model from carrying two
competing representations of editor intent.

The processor gains a normalized editor model that can serve record-based and
abstract-class form sources consistently. That model should be introduced
before adding several new annotations, otherwise each editor family will
duplicate type validation, descriptor generation, resource generation, and view
contract logic.

Typed option descriptors create one shared contract for combo boxes, radio
groups, checkbox groups, and custom selection editors. They keep value identity
separate from resource-backed labels, descriptions, disabled reasons, and
validation metadata, while still making that metadata visible to generated
tests.

Field dependencies make relevance and validation participation explicit. This
reduces handwritten listener code but requires clear rules for inactive values,
dirty-state aggregation, and generated diagnostics when dependencies point at
unsupported fields.

Component state models add another generated layer, but only for fields or
auxiliary regions that need dynamic presentation state. They keep enabled,
editable, visible, relevant, protected, busy, and validation-display state out
of the semantic value model. This should make advanced business forms easier to
test and less dependent on direct Swing component mutation.

Auxiliary component state models make non-field UI state first-class. Dialog
tabs, sections, button rows, and groups can participate in the same dependency,
busy, validation-summary, and presenter-control model as individual fields.
Because they are declared as annotated form members, generated accessors do not
appear from undocumented string ids or from a separate contract that must point
back to its owning form.

Picker and rich-value fields must keep summary rendering separate from the
semantic value, or they will recreate the invalid-intermediate-text problem in
a less explicit form.

Richer editor metadata increases release obligations. Generated names,
descriptor shapes, annotation attributes, and behavior-plan contracts become
public API once released and should be covered by generated-source tests and
examples before the release line is declared stable.

Some useful Swing controls will remain out of scope for built-in annotations.
That is acceptable only if the extension surface is strong enough for custom
business editors to participate in generated keys, resources, constraints,
semantic field models, summaries, and behavior plans.

## Implementation Progress

The implementation has already completed the first vertical slice rather than
only the preparatory model work. The implemented surface currently includes:

- `EditorDescriptor`, `EditorFamily`, and `SelectionShape` in core metadata,
  with generated field descriptors using structured editor descriptors for the
  existing text, date, combo-box, checkbox, and radio-group families;
- enum-backed option metadata through `@CoruscoForm.Option`, generated
  `OptionKey`, `OptionDescriptor<T>`, `OptionResourcePrefix`, and per-form
  `*Options` companions;
- source-level `@RadioGroup` processing for enum values, including generated
  view contracts, descriptors, resource-backed option lists, compile-time
  option-key diagnostics, tests, and an example;
- generated `*PresentationModel` companions that own opt-in field state models
  and auxiliary abstract form members returning `ComponentStateModel`;
- source-level `@CoruscoForm.DependsOn` rules for generated component state, with
  source-member validation, compile-time normalization of enum option keys,
  enum constants, and checkbox literals into typed generated `FieldDependency`
  companions, generated behavior-plan wiring, and Swing binding support for
  enabled, visible, and relevant state;
- abstract-class form sources are treated as source/owner contracts whose
  generated immutable result classes are standalone semantic value classes,
  while generated presentation models own auxiliary `ComponentStateModel`
  members;
- core form/table source annotations were renamed to `@CoruscoForm` and
  `@CoruscoTable`; Swing-specific generation is requested separately with
  package-level `@SwingCompanionPackage`;
- form and table metadata annotations are retained in class files so
  `@SwingCompanionPackage(forms = ..., tables = ...)` can generate Swing
  companions in adapter packages that depend on already compiled model
  classes.

The implemented dependency model is intentionally narrower than the full ADR
target. It uses source-level `CoruscoForm.DependencyEffect` values that are
normalized into generated core `DependencyEffect` metadata, but it does not yet
model inactive value policy, inactive validation policy, clearing/retaining
semantic values, logical `or`, dynamic option dependencies, or dependency
participation in commit semantics.

The implementation has not yet completed the later editor families:
numeric ranges, mapped and tri-state checkboxes, checkbox groups,
picker fields, rich opaque values, dynamic option sources, summary-renderer
contracts, or the custom editor extension API. The future option-source
annotation should avoid the generic `@Options` name; its exact shape is still
open.

## Out of Scope

This decision does not define a visual layout DSL for generated forms. Editor
families describe how a single semantic field can be edited, not where the
component appears in a screen.

This decision does not introduce annotation-driven parent/sub-form composition.
Multi-form dialog sessions are covered by ADR 202606.003.

This decision does not require Corusco to ship every concrete Swing widget.
Applications may still bind custom components to generated field models when a
domain-specific editor is not general enough for the library, but those custom
editors should use the same metadata and binding-extension model as built-in
editor families.

This decision does not change the basic text-field parse model. Text input that
needs invalid intermediate text should continue using `TextFieldModel`.

## Implementation Notes

1. Replace `FieldKind` in generated field descriptors with a structured
   editor-family descriptor. Model existing text, date, combo-box, and checkbox
   fields as descriptor instances instead of enum constants.
2. Introduce an internal normalized editor specification used by records and
   abstract-class form sources before adding new editor annotations.
3. Add a typed option descriptor contract for closed and application-provided
   option sets, including stable option identity, generated resource keys,
   disabled reasons, ordering, and validation behavior.
4. Add field dependency metadata for simple relevance rules, including inactive
   value/commit semantics, before wiring it to Swing visibility or enablement.
5. Introduce opt-in generated field component-state models that are attached to
   controls alongside normal field value models.
6. Add direct auxiliary component-state members for non-field regions such as
   groups, sections, dialog buttons, wizard pages, and tabs.
7. Define state-composition rules for descriptor defaults, dependencies,
   validation, presenter overrides, and transient busy/loading state.
8. Define the extension contract for application-specific editors: stable editor
   family id, semantic value type, optional option/range/picker metadata,
   summary rendering, component-state handling, and Swing binding behavior.
9. Implement enum radio groups end to end first: annotation/API surface,
   processor validation, generated descriptors/options/view contract, Swing
   binding behavior, generated-source tests, Swing binding tests, and one
   compiling example.
10. Implement numeric range metadata second, covering supported numeric source
   types, bounds, step size, slider/spinner intent, validation interaction, and
   generated diagnostics for invalid annotation combinations.
11. Add checkbox value mapping and nullable three-state support after the
   editor-family model can express semantic values separately from selected
   states.
12. Add enum checkbox groups for `EnumSet<E>` once option metadata and
   multi-selection binding can be represented without custom string ids.
13. Add picker fields after the semantic-value/display-summary/action split is
   part of the public editor descriptor API.
14. Update annotation package docs, generated-form package docs, manual examples,
   and processor diagnostics with each editor family rather than documenting the
   feature only in tests.
