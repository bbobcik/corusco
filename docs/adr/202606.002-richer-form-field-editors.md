# ADR 202606.002: Richer generated form field editors

Status: Proposed

Date: 2026-06-19

## Context

Generated form fields currently cover text fields, date text fields, combo
boxes, and boolean checkboxes. This is enough for the first generated-form
pipeline but not enough for practical business Swing forms.

Applications commonly need radio selectors, custom checkbox mappings,
checkbox groups, numeric sliders or spinners, and dialog-selected values such
as colors, fonts, files, directories, and certificates. Some of these values
can be summarized as text, but text is not their authoritative edit model.

This decision is separate from mixed generated form sources. It applies to both
record-based forms and abstract-class forms once abstract-class support exists.

## Decision

Separate field value semantics from editor family. The core model remains
based on semantic `FieldModel<O, T>` and text-preserving
`TextFieldModel<O, T>`. An annotation chooses the intended editor family and
metadata; Swing bindings decide how a specific component edits the semantic
value.

Add editor families incrementally:

- radio group: enum-backed and custom-option backed single selection;
- checkbox: boolean two-state, plus custom two-state or three-state value
  mapping;
- checkbox group: enum-backed `EnumSet<E>` first, custom set-like values later;
- numeric range editor: slider/spinner style fields backed by numeric semantic
  values rather than raw text;
- picker fields: color, font, file, directory, and other dialog-selected
  values;
- opaque rich values: certificate-like values where a compact textual display
  is a summary, not the edit representation.

The metadata model should grow from a small fixed `FieldKind` enum into a
stable editor-family model that can carry option-source and value-mapping
metadata. The processor should continue generating typed companions rather
than requiring users to write string identifiers by hand.

Picker-style fields should bind to semantic values. A text field or label may
display a summary, but the summary is not the authoritative form value.

## Consequences

New field editors can share typed keys, descriptors, validation, resources, and
generated behavior plans instead of becoming ad-hoc Swing code.

Generated form metadata becomes a better description of intended presentation
families rather than a small list of Swing component classes.

Expanding editor metadata can destabilize `FieldKind` users unless migration is
staged carefully.

Picker and rich-value fields must keep summary rendering separate from the
semantic value, or they will recreate the invalid-intermediate-text problem in
a less explicit form.

## Implementation Notes

1. Decide whether `FieldKind` evolves directly or is superseded by a richer
   editor-family descriptor.
2. Add one editor family end to end, preferably enum radio group or numeric
   slider.
3. Add generated-source tests, Swing binding tests, and one compiling example.
4. Add custom option or value-mapping support after the enum/simple case is
   stable.
5. Add picker fields after the semantic-value/editor-summary split is visible
   in public API.
